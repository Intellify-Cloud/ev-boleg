Footer | For Jekyll | 21Dev    # Reusable Jekyll Footer Design   This document describes a reusable footer pattern for Jekyll websites and coding agents. It captures the design, architecture, and implementation rules without depending on a particular brand, industry, or JavaScript framework.   ## Design intent   The footer is a spacious, modern information area built around clarity and useful navigation. It should feel calm and intentional rather than like a dense sitemap.   The layout has two levels:   1. A main area containing the brand, a short description, optional social links, and configurable navigation columns. 2. A quiet bottom strip containing copyright, legal links, optional partner attribution, and an optional website creator credit.   Use generous white space, subtle borders, muted supporting text, strong section labels, and the site's existing accent colour for interactive states.   ## Technology rules   - Implement the footer with Jekyll Liquid, semantic HTML, YAML data, and Sass. - Do not introduce React, Tailwind, TypeScript, icon packages, or component frameworks solely for the footer. - Reuse the project's existing design tokens, layout utilities, and icon system where available. - Keep content separate from presentation so links and labels can be updated without editing the template. - Adapt the pattern to the existing project rather than forcing a new technology stack into it.   ## File responsibilities   ### `_includes/footer.html`   Owns the semantic footer markup and Liquid loops.   - Use a single `<footer>` landmark with a stable ID or class. - Link the logo to the home page using `relative_url`. - Render navigation columns from a YAML collection such as `site.data.sitetext.footer.sections`. - Render social links only when they are configured. - Mark external links with `target="_blank"` and `rel="noopener"` when opening a new tab is appropriate. - Give navigation groups descriptive `aria-label` attributes. - Keep partner and creator attribution separate from the site's canonical URL configuration. - Keep the footer in normal document flow.   Avoid scripts that force the footer to `position: fixed`. A large responsive footer can overlap content on short pages when removed from normal flow.   ### Footer data   Store editable content in a YAML data file such as `_data/sitetext.yml` or `_data/footer.yml`. A generic structure could look like this:   ```yaml footer:   copyright_prefix: "Copyright"   description: "A concise description of the organisation and its value."   privacy_label: "Privacy Policy"   privacy_url: "/privacy"   terms_label: "Terms and Conditions"   terms_url: "/terms"   social_label: "Social links"     sections:     - title: "Explore"       links:         - text: "About"           url: "/about"         - text: "Services"           url: "/services"       - title: "Resources"       links:         - text: "Help Centre"           url: "/help"         - text: "Articles"           url: "/articles"       - title: "Get in touch"       links:         - text: "+00 000 000 0000"           url: "tel:+000000000000"           external: true         - text: "hello@example.com"           url: "mailto:hello@example.com"           external: true     social:     - title: "LinkedIn"       url: "https://www.linkedin.com/company/example"       icon: "fab fa-linkedin-in"     credits:     - text: "Powered by Example Partner"       url: "https://partner.example.com"     - text: "Website by Example Studio"       url: "https://studio.example.com" ```   Internal URLs must pass through Liquid's `relative_url` filter. Absolute web addresses and protocol links such as `tel:` and `mailto:` should bypass that filter. A flag such as `external: true` can communicate this distinction to the template.   ### `_assets/layout/_footer.scss`   Owns all footer presentation. Adjust the exact path to match the project's Sass organisation.   The recommended class structure is:   `text .footer ├── .footer-main │   ├── .footer-brand │   │   ├── .footer-logo │   │   ├── .footer-description │   │   └── .footer-social (optional) │   └── .footer-navigation │       └── .footer-column └── .footer-bottom     ├── .footer-copyright     ├── .footer-legal     └── .footer-credits (optional) ```   Use existing Sass variables for colours, spacing, typography, motion, and breakpoints. Do not duplicate literal values when a project design token already exists.   ## Liquid implementation pattern   Keep repeated links data-driven:   ```liquid <nav class="footer-navigation" aria-label="Footer navigation">   {% for section in site.data.sitetext.footer.sections %}     <div class="footer-column">       <h2>{{ section.title }}</h2>       <ul>         {% for link in section.links %}           <li>             <a href="{% if link.external %}{{ link.url }}{% else %}{{ link.url | relative_url }}{% endif %}">               {{ link.text }}             </a>           </li>         {% endfor %}       </ul>     </div>   {% endfor %} </nav> ```   Add new-tab attributes conditionally rather than applying them to every link. Do not open internal navigation, telephone links, or email links in new tabs without a clear reason.   ## Responsive behaviour   Use a mobile-first layout.   - Mobile: one brand block followed by one or two navigation columns per row; remaining columns wrap naturally. The bottom strip stacks and centres. - Tablet: distribute navigation into two or three columns and allow the bottom strip to become horizontal when space permits. - Desktop: place brand content on the left and navigation columns on the right using CSS Grid. - Let the number of navigation columns respond to available space rather than assuming exactly three. - Allow long email addresses and labels to wrap without causing horizontal scrolling. - Keep touch targets comfortably spaced.   Example breakpoint strategy:   ```scss .footer-main {   display: grid;   gap: var-or-token-for-large-space; }   .footer-navigation {   display: grid;   gap: var-or-token-for-medium-space;   grid-template-columns: repeat(2, minmax(0, 1fr)); }   @media (min-width: 768px) {   .footer-navigation {     grid-template-columns: repeat(3, minmax(0, 1fr));   } }   @media (min-width: 992px) {   .footer-main {     grid-template-columns: minmax(280px, 0.9fr) minmax(480px, 1.1fr);   } } ```   Replace placeholder spacing references and breakpoints with the project's own tokens.   ## Visual details   - Use a neutral background and a subtle top border to distinguish the footer from page content. - Preserve the logo's intrinsic aspect ratio and set a consistent maximum height. - Limit the brand description to a comfortable reading width and render it in supporting text colour. - Keep column headings compact and visually stronger than their links. - Use muted navigation links with a clear hover and focus treatment based on the brand accent. - Separate the bottom strip with a thin border and slightly smaller typography. - Make creator or partner credits legible but visually quieter than primary navigation. - Avoid excessive shadows, oversized icons, and decorative elements that compete with the content.   ## Accessibility requirements   - Provide meaningful logo alt text. - Use footer and navigation landmarks correctly. - Keep heading levels logical within the surrounding page. - Do not use headings purely to achieve a visual size. - Provide visible keyboard focus styles. - Include accessible labels for icon-only social links. - Preserve sufficient text, border, and interaction-state contrast. - Avoid opening internal links in new tabs. - Ensure keyboard and touch users can comfortably activate every link. - Verify that zooming and narrow viewports do not clip content.   ## Domain and attribution rules   Treat these concepts as separate configuration values:   - Canonical website URL - Partner or platform URL - Website creator or agency URL   Partner and creator links are footer credits. They must never replace the canonical host used by Jekyll, metadata, sitemap generation, or structured data.   Example:   ```yaml url: https://www.example.com ```   ```yaml footer:   credits:     - text: "Powered by Partner"       url: "https://partner.example.com"     - text: "Website by Studio"       url: "https://studio.example.com" ```   ## Build process   Use the build pipeline already defined by the project. For a project where Webpack compiles Sass before Jekyll runs, the sequence may be:   ```powershell npm run bundle bundle exec jekyll build --trace ```   For local review:   ```powershell bundle exec jekyll serve --host 127.0.0.1 --port 4000 ```   Inspect both a normal content page and a short page such as the custom 404 page. A short page is especially useful for exposing footer overlap and spacing problems.   ## Agent verification checklist   Before handing off a footer change, verify all of the following:   - The asset and Jekyll builds complete without errors. - The generated HTML contains the expected footer sections and configured credits. - Internal links respect `relative_url`. - New-tab external links use `rel="noopener"`. - The footer remains in normal document flow and does not overlap content. - Desktop, tablet, and mobile layouts have no horizontal overflow. - The logo remains legible and correctly proportioned. - Contact, resource, legal, and navigation links resolve correctly. - The canonical site URL remains unchanged unless the task explicitly requires a domain change. - The compiled stylesheet includes the updated footer rules. - Empty optional sections do not render blank containers. - Unrelated generated files are not committed merely because local tooling changed formatting or line endings.   ## Reuse guidance   Copy the structural approach, not the example content. Replace the logo, description, navigation sections, contact details, legal links, social profiles, and credits through YAML. Preserve the semantic hierarchy, responsive grid, accessible link treatment, normal document flow, and restrained bottom strip.   When adapting the pattern, inspect the destination project first. Match its data naming, tokens, breakpoints, build commands, and accessibility conventions instead of assuming this exact file structure exists everywhere.
